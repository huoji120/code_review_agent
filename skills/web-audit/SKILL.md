# Web 后台审计 Skill

当目标是 Web 时加载这个 skill。同时如果目标是java的web也需要精细化加载java的skills

判定门槛：

- 如果一个问题必须管理员、高权限运营、超级管理员或已经拥有危险后台权限的账号登录后才能触发，而且本质上只是“管理员自己能做危险操作”，通常不算漏洞。
- 如果普通用户、低权限用户、游客、未授权访问者，或者仅凭可注册账号/弱前置条件就能触发，则属于漏洞候选，应继续审计利用链和实际影响。
- 如果攻击前提是“拿到管理员账号后可执行某动作”，而这个动作本来就是管理员职责范围内能力，不要当成漏洞上报。
- 如果低权限用户能越权进入管理员功能、调用管理员接口、绕过租户隔离、篡改关键配置、导出敏感数据、写入模板/脚本/文件，则是漏洞。

重点场景：
- 优先寻找sql注入,XSS这种经典问题,特别是SQL注入,先排除经典的漏洞.xss有一点需要注意,当你发现某个接口/函数疑似没做XSS过滤的时候,你需要寻找到用这个函数谁调用的,XSS需要调用链完整,从入库到入库都需要有证据
- jwt/数据库明文存储除非有任意文件读取否则都是正常行为不需要报
- 超级管理员能访问一切资源是正常的
- 认证与授权：未授权访问、越权、水平/垂直权限绕过、菜单鉴权与接口鉴权不一致、前端限制后端未校验、租户隔离失效。
- 登录与找回：验证码可被绕过或刷爆、短信/邮件验证码无限尝试、密码重置链接可预测、找回流程身份绑定不足、图形验证码只在前端校验。
- 用户与角色管理：低权限用户给自己加角色、修改组织/部门/租户字段、编辑他人资料、重置他人密码、批量导入导出越权。
- 文件上传下载：任意文件上传、上传到可执行目录、文件名/路径穿越、附件鉴权缺失、下载接口直接拼接路径、临时文件泄露。
- 配置与代码生成：模板编辑、代码生成、脚本执行、SQL 执行、定时任务、Webhook、插件安装、导入导出、备份恢复等高危后台能力。
- 审批与业务流：重复提交、状态竞争、越级审批、撤回/驳回绕过、支付金额/折扣/库存/积分等关键字段可篡改。
- 逻辑漏洞：业务规则绕过、次数/额度限制缺失、领取/提现/下单/退款重复触发、活动资格校验缺失、邀请返利作弊、验证码/短信接口被刷。
- 风控与限流：验证码重放、接口无频率限制、批量枚举用户名/手机号/订单号、登录爆破、短信轰炸、邮件轰炸、图形验证码形同虚设。
- 后台常见危险点：富文本、模板、通知内容、导入文件、报表过滤器、搜索器、代码预览、主题编辑、在线 SQL、日志查看、对象存储回调。

审计要求：

- Web 后台往往涉及多个组件联动，例如控制器、服务层、DAO、模板、上传目录、对象存储、定时任务、消息队列、配置中心、代码生成器、审批流和租户/组织体系。不能只看单个文件就下结论。
- 必须进行多文件联动审计，优先从入口 -> 鉴权/业务校验 -> 数据传播 -> 最终危险操作的完整链路分析。要多用 `variable_review_update` 和 `flow_review_update` 追踪关键变量、权限标记、对象归属、文件路径、状态字段和跨文件调用链。
- 对怀疑漏洞的点，提交前必须再次用变量排查和 flow 排查做二次验证，补齐入口、传播、缺失校验、sink 和实际影响证据，不能只凭单文件直觉提交漏洞。
- 必须区分“管理员本来就有的危险能力”和“低权限/未授权用户不该有却拿到了的能力”。
- 必须说明攻击者身份边界：未登录、普通用户、业务用户、运营、管理员、超级管理员。
- 对验证码、短信、邮件、找回、导入导出、审批流、代码生成、模板编辑、文件上传等场景，要同时检查接口权限、频率限制、对象归属校验、状态机校验和最终危险 sink。
- 逻辑漏洞必须写清楚业务前置条件、步骤、可重复性、收益或破坏结果。不能只写“可能存在逻辑问题”。
- 如果某问题只有管理员自己登录后台后才能触发，且没有越权、提权、账号接管、租户突破、持久化危害或对其他用户/系统边界的实际破坏，默认不要报漏洞。
- 先实现至少10个计划,再做其他的审计
- 审计所有插件的 Action/Widget 类
- 在审计计划阶段必须搜索 factory 等注册模式，逐一排查这些，不可忽略
- 如果网站有插件,必须逐个审计插件的类的鉴权链路和CURD机制链路,不可批量忽略,原因是插件大部分是个人作者写的,质量比较低下,比如wordpress和typecho或者discord的插件是重灾区
- 注意任意文件上传,以及sql加SSRF,不要随便skipping文件,PHP的CMS需要经常跨链路检查
